# 对比实验：面经 RAG + 智谱大模型 vs DeepSeek/ChatGPT 直接上面经+简历

本文档用于验证「**面经库 RAG + 智谱大模型**」在**简历深挖**与**面试对话**上的**差异性与优势**，并与「**把面经+简历整段上传给 DeepSeek 或 ChatGPT**」进行对比。  
两者都会调用大模型生成内容；差别在于：本系统先按字段分块检索面经库，再拼成结构化 prompt 给智谱；对照组则直接把同一份面经+简历全文交给通用模型。

**说明**：若本地未配置智谱 API Key，后端无法启动，此前对比实验只做了 RAG 单元测试（不调用任何大模型）。要跑完整对比，请在本地配置 Key（**为安全请勿在对话里粘贴 Key**，使用环境变量或 `application-local.yml`），然后按下方流程执行。

---

## 〇、「直接上传面经+简历到 DeepSeek/ChatGPT 不就行了，落库+RAG 有什么优势、是不是多此一举？」

直接上传面经图片和简历到大模型网页对话，确实能生成不错的深挖题和模拟面试，**但和「面经落库 + 向量检索 + RAG 结合大模型」是不同场景、不同能力**，并不是多此一举，而是各有优劣。

| 维度 | 直接上传（DeepSeek/ChatGPT 等） | 面经落库 + RAG + 大模型（本系统） |
|------|----------------------------------|-----------------------------------|
| **使用方式** | 每次新公司/新岗位：重新找面经、重新上传图片+简历，对话关掉就没了 | 面经**一次录入、永久可检索**；换公司/部门只改下拉框，同一份简历反复用；面经库越积越多，检索越准 |
| **检索与精准度** | 模型看到的是「你塞进去的全文」，无法按公司/部门/题型筛选，容易混用不同公司考点 | **按公司、部门、题型**（实习/项目/八股/大模型/算法）检索，只把「当前目标」相关面经片段喂给模型，出题更贴**该公司该岗位**真实考点 |
| **结构化与可维护** | 图片/PDF 非结构化，无法单独增删改某条面经，也难以做「字节基础架构常问什么」这类统计 | 面经**结构化存储**，可编辑、可删除、可按公司/部门/类型查询；后续可做高频考点、面经热度等分析 |
| **数据与隐私** | 简历和面经都进大模型厂商的对话上下文，可能被留存或用于训练 | 面经和向量检索在**你自己的服务**里，只把检索摘要+简历文本发给大模型，数据可控、可脱敏 |
| **可复现与迭代** | 同一份简历+面经，不同会话/模型版本结果可能差异大，难以做 A/B 或对比实验 | 同一面经库+同一检索逻辑，生成结果更**可复现**，便于迭代 prompt、检索策略和做对比实验 |
| **适用场景** | **单次、临时**：随便试几家、不想搭服务、零配置上手即用 | **多公司、多岗位、长期备战**：面经越攒越多，希望「换一家公司就换一拨题」且题目贴真实面经 |

**一句话**：直接上传适合「偶尔面几家、图省事」；面经库 + RAG 适合「认真备战、多投多面、希望题目可复用、可检索、按目标公司精准出题且数据在自己手里」的求职者。两者可以**并存**：临时用网页对话快速试水，长期用本系统积累面经、做针对性深挖和对话。

## 〇-2、如何证明「优化 chunk 和 embedding 后的 RAG」比之前有优化？

1. **逻辑自证（已落地）**  
   - 运行单元测试：`mvn test -Dtest=RagServiceTest`。  
   - 测试验证：面经按**字段分块**（实习/项目/八股_Java/八股_AI/算法）写入向量库，**结构化检索** `searchStructuredForDeepQuestions` 能按类型聚合返回，且**公司/部门过滤**生效。即「分块 + 类别感知检索」行为正确。
2. **效果自证（需本机配置 API Key 后跑一遍）**  
   - 按第三节启动本系统，用**同一份简历+公司/部门**生成深挖问题；  
   - 按第四节将**同一份面经+简历**贴给 ChatGPT/DeepSeek，用附录 B 话术生成题目；  
   - 按第六节评估表对两组输出打分，即可证明「面经库 RAG + 结构化 prompt」在**题目针对性、与面经贴合度、无废话**等维度上的差异性与优点。

## 一、本系统 RAG 优化点（已实现）

| 优化项 | 说明 | 实现位置 |
|--------|------|----------|
| **按字段分块 (Field-based Chunk)** | 一条面经拆成多块：总述、实习、项目、八股_Java、八股_AI、算法；每块带 type/company/department/experienceId | `RagService.buildChunks()` |
| **类别感知检索** | 深挖问题时按类型聚合，拼成【参考面经-实习】【参考面经-项目】等，模型按块对应出题 | `RagService.searchStructuredForDeepQuestions()` |
| **同一面经限流** | 同一 experienceId 最多取 2 块，避免单条面经占满结果 | `MAX_CHUNKS_PER_EXPERIENCE` |
| **公司/部门过滤** | 检索时按 company/department 过滤，只取目标公司/部门面经 | `search()` / `searchStructuredForDeepQuestions()` |
| **结构化 Prompt** | System：大厂面试官、输出格式（1.2.3. 开头、实习→项目→八股→大模型）、禁止开场白/总结；User：分块给参考面经+简历 | `InterviewAgentService.InterviewCoach` |
| **简历对齐** | Prompt 中明确「针对简历中的经历和技术栈出题，不要问简历未涉及的方向」 | 同上 |
| **Embedding 可选** | 可配置智谱 Embedding-2 提升中文检索（未配置则本地 AllMiniLM） | `application.yml` / `ZhipuEmbeddingModel` |

## 二、实验设计

### 2.1 统一输入（便于复现）

- **目标公司**：字节跳动  
- **目标部门**：基础架构  
- **简历**（节选，固定用于对比）：

```
教育：某 985 计算机本科，2025 届。
实习：字节跳动 基础架构 实习 6 个月，参与 KV 存储引擎开发，使用 RocksDB，做过 compaction 优化与监控告警。
项目：基于 Redis 的分布式缓存课程设计；大模型 RAG 检索小项目，使用 LangChain + 智谱 API。
技能：Java、Spring、Redis、MySQL、RAG/大模型基础。
```

- **面经素材**（用于「无 RAG」对照组时整段粘贴给 ChatGPT/DeepSeek）：见下方「附录：面经样本」。

### 2.2 实验组

| 组别 | 方式 | 说明 |
|------|------|------|
| **A. 本系统（RAG + 智谱）** | 在本系统「AI 面试模拟」选择公司=字节跳动、部门=基础架构，粘贴上述简历，点击「生成深挖问题」；面试对话在「与面试官探讨」中进行 | 后端先**检索面经库**（按字段分块、按类型聚合），再拼成**结构化 prompt** 给**智谱大模型**生成深挖题与追问/答疑 |
| **B. DeepSeek / ChatGPT（无 RAG）** | 将「附录 A 面经样本」+ 上述简历 **整段粘贴**到 DeepSeek 或 ChatGPT，用附录 B/C 话术分别做「深挖出题」和「面试对话」 | 模型仅基于你给的**全文**生成，无面经库检索、无按类型分块，可作为对照组 |

### 2.3 评估维度（1–5 分，5 最好）

| 维度 | 说明 |
|------|------|
| **题目针对简历** | 是否明显针对简历中的实习/项目/技术栈（RocksDB、Redis、RAG、LangChain 等）出题，而非泛泛的「介绍一下你的项目」 |
| **顺序与结构** | 是否大致按 实习→项目→八股→大模型 顺序，且每题简洁（约 50 字内） |
| **与面经贴合度** | 题目风格、考点是否与该公司/部门真实面经一致（如基础架构常问存储、一致性、性能） |
| **无废话/无超纲** | 是否无开场白、无总结、不问简历未涉及的方向 |
| **可追问性** | 题目是否具体到便于面试官下一轮追问（如「RocksDB compaction 如何调优」而非「你做过什么优化」） |

## 三、一键对比流程（RAG+智谱 vs DeepSeek/ChatGPT）

按顺序执行即可完成「深挖 + 面试对话」两项对比。

1. **配置智谱 API Key**  
   - 环境变量：`export ZHIPU_API_KEY=你的Key`  
   - 或在本项目 `src/main/resources/` 下新建 `application-local.yml`，内容：`zhipu:\n  apiKey: "你的Key"`（该文件已在 .gitignore，不会提交）

2. **确保本系统有面经可检索**  
   - 启动后端+前端（见 3.1），在「面经搜索」页添加至少一条「字节跳动-基础架构」面经（可直接用附录 A 的内容录入），这样 RAG 才有参考内容。

3. **本系统 · 深挖**  
   - 浏览器打开「AI 面试模拟」→ 公司=字节跳动、部门=基础架构、简历=第二节简历 → 点击「生成深挖问题」。  
   - 或将后端启动后执行：`./scripts/run_rag_experiment.sh`，结果在 `docs/experiment_output_rag.txt`。  
   - 将**完整输出**复制到本文「六、本系统输出（A 组）」。

4. **DeepSeek/ChatGPT · 深挖**  
   - 打开 DeepSeek 或 ChatGPT，先粘贴**附录 A 面经样本**全文，再粘贴**第二节简历**全文。  
   - 再发送**附录 B** 的提问话术。  
   - 将模型生成的题目复制到「七、通用模型输出（B 组）」。

5. **本系统 · 面试对话**  
   - 在「与面试官探讨」中任选一道深挖题，输入简短回答或输入「请给参考答案」。  
   - 观察回复是否结合面经考点、是否像真实面试官追问。可截一段对话贴到「九、本系统面试对话示例」。

6. **DeepSeek/ChatGPT · 面试对话**  
   - 在同一对话里，复制一道深挖题，发送**附录 D** 的追问/答疑话术。  
   - 对比其回复是否贴合你给的面经和简历。可截一段贴到「十、通用模型面试对话示例」。

7. **填写评估表**  
   - 根据「六、结果记录表」对两组深挖输出打分并写结论；面试对话可简要写在本系统 vs 通用模型谁更贴面经、谁更易泛泛而谈。


```bash

# 或指定后端地址：./scripts/run_rag_experiment.sh http://localhost:8080/api
```

结果会写入 `docs/experiment_output_rag.txt`，便于与通用模型输出对比。
