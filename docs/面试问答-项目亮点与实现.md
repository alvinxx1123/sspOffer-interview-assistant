# 面试官常问：项目实现与亮点（回答要点）

基于 sspOffer 面经助手当前代码整理，便于面试时结合项目回答。

---

## 1. 项目里有做压缩上下文吗？上下文窗口过大会有什么问题，该怎么解决？

**有做。**

- **对话记忆**：`InterviewChatService` 里用 **MessageWindowChatMemory.withMaxMessages(20)**，只保留最近 20 条 user/assistant 消息，避免无限涨。
- **会话恢复**：从 DB 恢复时只灌回 **最近 20 条**（`MAX_RESTORE_MESSAGES`），不是全量历史。
- **单条长度限制**：简历放进 system 时超过 **2000 字会截断**；RAG 注入面经时单条超过 **500/600 字截断**；RAG 检索结果拼进 prompt 时每段也做了长度截断（如 600 字 + "..."）。
- **RAG 注入量**：每轮只检索 **top 5** 条面经片段注入，不是整库塞进去。

**上下文过大的问题**：超出模型 context 上限会截断或报错、延迟和 token 成本升高、无关信息干扰回答质量。

**解决思路**：限制条数（滑动窗口）、单条截断、按需 RAG 只注入当轮相关片段；若后续要支持更长对话，可以加摘要压缩（如对较早消息做 summarization 再塞进 context）。

## 2. 流式返回怎么做的？用的 SSE 和 WebFlux 吗？

**深挖问题流式**：用的是 **SSE（Server-Sent Events）**，**没有用 WebFlux 做流式**。

- **实现**：Spring MVC 的 **SseEmitter**，`produces = MediaType.TEXT_EVENT_STREAM_VALUE`。接口里 `new SseEmitter(120_000L)`，在**单线程 Executor** 里调 `agentService.generateInterviewQuestionsWithSteps(...)`，通过回调 `step -> emitter.send(SseEmitter.event().name("step").data(step))` 先发多条「思考过程」事件，最后 `emitter.send(SseEmitter.event().name("result").data(result))` 发完整结果。
- **协议**：HTTP 长连接，`Content-Type: text/event-stream`，前端用 **fetch + ReadableStream** 读 body，按行解析 `event:` / `data:`，分别处理 step 和 result。

**WebFlux 在项目里的作用**：用来做**异步 HTTP 客户端**（WebClient）调 Piston 代码执行 API，**不是**用来做流式响应的。流式响应是 **Servlet 栈的 SseEmitter** 实现的。

## 3. 给出结果是怎么给的？

- **深挖问题流式接口**（`/questions/stream`）：先发多条 **step** 事件（如「正在检索面经…」「检索结果：实习 有参考…」「正在生成…」「生成完成」），再发一条 **result** 事件（完整题目文本）。前端边收边更新「思考过程」区域，最后用 result 更新题目列表。
- **非流式接口**（如 `/questions`）：普通 **JSON 响应**，一次返回整段生成的题目文本。
- **面试对话**（`/chat-session`）：同步调用大模型，返回 **JSON**，如 `{ "reply": "..." }`。

总结：流式用 **SSE 多事件（step + result）**，非流式用 **单次 HTTP JSON**。

## 4. 召回精度不高，怎么提升？

**当前做法**：按字段分块（实习/项目/八股/大模型/算法）、公司/部门过滤、同一条面经最多取 2 块、相似度阈值 0.4、AllMiniLM 或智谱 Embedding。

**可说的提升方向**（部分在《优化方案-面经RAG与对话升级》里也有）：

1. **换更好的 Embedding**：智谱 Embedding-2、BGE 等中文更强的模型，提升语义匹配。
2. **加 Reranker**：检索先拿 topK（如 20），再用 Reranker 对 query+候选精排，取 top 5～10 再给大模型，精度会明显提升。
3. **Query 扩展**：用户只选公司/部门时，query 里拼上「实习 项目 八股 大模型 算法」等关键词，提高召回覆盖。
4. **分块策略**：单块长度（当前 800 字）、是否按句子/段落再切、overlap 等，按效果调；保证 metadata（type、company、department）准确便于过滤。
5. **数据**：更多高质量、标注或清洗过的面经，召回上限会提高。

## 5. AI 生成代码不是你想要的怎么办？有什么提效方式？

**项目里**：在线 IDE 是**用户自己写代码**，后端用 Piston 执行，**不是**“AI 自动生成代码再执行”。和“生成代码不满意”更相关的是：**智能助手**里模型可能返回代码片段、或用户让模型“写一段某某代码”。

**若面试官问的是「AI 生成代码不符合预期」的通用提效**，可以答：

- **Prompt**：更具体的需求描述、约束（语言、框架、接口），加 **few-shot** 示例，让模型按示例风格和结构生成。
- **分步生成**：先生成思路/伪代码，确认后再生成具体代码，减少一次性生成整段不对的情况。
- **人工编辑**：生成结果在 IDE 里可编辑，用户改完再运行，形成「生成 → 修改 → 运行」闭环。
- **模板/约束**：对 Java 多类、无 main 等情况，我们做了**入口识别和类顺序归一化**，保证 Piston 按正确入口执行，减少“跑不起来”带来的无效重试。

**项目里可说的提效**：入口识别 + 类顺序处理、超时与错误信息返回、支持自建 Piston 解决公网超时，都是让“代码执行”环节少踩坑、快速拿到结果。
