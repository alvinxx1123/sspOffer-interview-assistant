根据简历「项目经历」模拟字节、腾讯等技术面试官的提问，并给出可参考的回答。建议先自己口头答一遍，再对照参考答案查漏补缺。

## 一、开场：项目介绍

**面试官**：先简单介绍一下你这个「sspOffer 面试助手」项目，你主要负责哪几块？

**参考答案**：  
这是一个面向互联网求职的 AI 面试准备平台，支持面经检索、AI 面试模拟、面试复盘、在线算法 IDE 和简历/投递管理。我主要负责后端整体：一是用 LangChain4j 做 RAG，把面经向量化并做相似度检索，再结合智谱 GLM 生成深挖题；二是把「查面经、查算法题、运行代码」等封装成 Tool，通过 Function Calling 让大模型按用户意图自动调用；三是在线 IDE 依赖的代码执行服务，对接 Piston；四是面经、简历、投递的 CRUD、管理员鉴权，以及部署到阿里云。上线后公网 Piston 超时，我改成 Docker 自建 Piston 并写了部署文档。

## 二、RAG 与 AI 面试模拟

**面试官**：RAG 在你项目里具体是怎么做的？从用户点「查面经」到出题，完整流程讲一下。

**参考答案**：  
面经先存到 H2，同时把公司、部门、岗位、内容等拼成一段可检索文本，用 LangChain4j 的 EmbeddingModel（我们用的 AllMiniLM）做向量化，写入 EmbeddingStore，目前是 InMemoryEmbeddingStore。检索时：用户选公司/部门或输入问题，先对 query 做 embedding，再用 `embeddingStore.findRelevant` 做相似度检索，取 topK 条面经文本，拼进大模型的 System/User Prompt，调用智谱 GLM 根据这些面经和用户简历生成深挖问题。启动时因为向量在内存里会丢，所以用 CommandLineRunner 从 DB 拉全量面经，通过 @Async 异步重建索引。

**面试官**：为什么用 InMemory 存向量？生产环境你会怎么改？

**参考答案**：  
当前是个人/小规模使用，数据量不大，内存实现简单、无额外依赖、检索也快。缺点是重启后向量丢失，所以启动时全量重建。生产上会换成持久化向量库，比如 Redis 存向量、或者 pgvector、Milvus 这类专门向量库，并做增量索引，而不是每次全量重建。

**面试官**：面经批量上传时为什么要异步建索引？

**参考答案**：  
批量上传如果同步做 embedding 和写入向量库，请求会拖很久，容易 502 或超时。所以保存到 DB 后只触发一次 `@Async` 方法，在后台线程里做向量化和索引，接口立刻返回。异步里要打好日志，避免静默失败，后续可以加监控告警。

**面试官**：为什么选智谱 GLM？有没有对比过其他模型？

**参考答案**：  
当时主要考虑国内可访问、有开放 API、文档清晰，智谱 GLM 满足这些，且对中文面试场景效果不错。没有在项目里系统对比其他模型；若被追问，可以说后续可以接入 OpenAI/Claude 等做 A/B 或按场景选模型。

## 三、Function Calling 与 Tool

**面试官**：Function Calling 是怎么实现的？大模型怎么知道该调哪个工具？

**参考答案**：  
我们用 LangChain4j 的 AiServices 把一组 Tool 注册给一个 ChatModel。Tool 就是用 `@Tool` 注解的方法，每个方法有一个描述字符串，例如「按公司或部门检索面经」「运行一段代码并返回执行结果」等。大模型根据用户输入和这些描述做意图识别，决定调用哪个工具、传什么参数；调用完后把工具返回结果再塞回对话，由模型生成最终回复。实现上有一个 `InterviewAssistantTools` 类，里面定义了查面经、查算法题、按 ID 取题、运行代码等几个 @Tool 方法，和 `InterviewAgentWithToolsService` 里的 Agent 绑定。

**面试官**：如果用户说「给我一道算法题」，后端会走哪条链路？有没有绕过模型？

**参考答案**：  
会先判断是否是「要一道算法题」这类意图；若是，可以走两条路：一是让模型通过 Tool 调用「获取题库列表/随机抽题」工具，工具返回题目和链接，模型再组织成回复；二是我们在后端做了短路逻辑，识别到这类请求后直接走本地题库随机抽题，用固定模板拼好「题目描述 + 本站在线 IDE 链接」返回，不经过大模型，响应更快、也避免模型把链接包成 HTML 导致前端跳转异常。所以实际是「算法题请求优先走后端直出，其它再走 Function Calling」。

**面试官**：为什么用 LangChain4j 而不是 Spring AI？

**参考答案**：  
做这个项目时 Spring AI 刚起来，LangChain4j 的 RAG、EmbeddingStore、Tool 文档更全，和智谱的对接也有现成示例，所以选了 LangChain4j。Spring AI 也是基于类似思路（ChatClient + Function Calling），若新项目从零选型，可以再对比两者在生态、与 Spring 集成度上的差异。

## 四、在线算法 IDE 与代码执行

**面试官**：在线 IDE 的代码执行是怎么实现的？为什么用 Piston？

**参考答案**：  
后端不直接执行用户代码，而是把代码、语言、stdin 等信息通过 HTTP 发给 Piston。前端发 POST 到我们 `/api/execute`，后端用 WebClient 再 POST 到 Piston 的 `/execute`，拿到 stdout、stderr、exitCode 后封装返回前端。选 Piston 是因为开源、支持多语言、有现成 API，避免自建沙箱和隔离环境；缺点是公网实例从国内访问容易超时，所以我们线上改成 Docker 自建 Piston，本机调用延迟低。

**面试官**：为什么采用 Monaco 编辑器 + Piston 代码执行这套方案？

**参考答案**：  
**Monaco 方面**：Monaco 是 VS Code 同款编辑器内核，开源、支持多语言高亮和补全，和力扣、牛客等主流 OJ 的编辑体验一致，用户无需适应。用 React 集成 `@monaco-editor/react` 即可，不需要自己造轮子。  
**Piston 方面**：需要在线执行多语言代码（Java、Python、Go 等），自建沙箱要维护多语言环境、隔离和资源限制，成本高；Piston 提供统一 HTTP API、多语言多版本、容器内执行，接入简单。  
**组合**：前端 Monaco 负责编辑和展示，后端只做转发和超时控制，执行完全交给 Piston，职责清晰；后续若要换执行引擎，只要改后端调用的服务，前端不用动。

**面试官**：为什么用 Docker 自建 Piston，而不是继续用公网 API？

**参考答案**：  
上线后发现从国内访问公网 Piston（如 emkc.org）经常超时或连不上，导致「运行」一直转圈、前端报 Failed to fetch，本质是网络延迟和稳定性问题。自建 Piston 的好处：① **延迟低**：后端和 Piston 在同一台 ECS 上，走 127.0.0.1，毫秒级；② **可控**：不依赖第三方可用性，自己决定超时、语言版本和资源；③ **Docker 部署简单**：官方提供镜像，一条 docker run 就能起服务，通过环境变量把 `PISTON_API_URL` 指到本机即可。代价是多占一点服务器资源，但对我们单机部署是可接受的，所以选了 Docker 自建。

**面试官**：Java 代码有多类、或者没有 main 方法，你怎么处理？

**参考答案**：  
Piston 执行 Java 时以「第一个类」为入口。我们先用正则看代码里有没有 `static void main`：如果没有，就自动生成一个 Main 类，main 里 new Solution() 并调常见题方法（如 twoSum）；如果有，就找到带 main 的类，把该类在源码里的顺序挪到最前面（保留 import 等），再发给 Piston，并保证文件名带 .java 后缀，这样入口就对了。

**面试官**：线上出现过「Host is not specified」「Failed to fetch」，你是怎么排查和解决的？

**参考答案**：  
「Host is not specified」说明请求 Piston 时 URL 为空，多半是环境变量 PISTON_API_URL 没在服务器生效（没写进 env 或没重启服务）。「Failed to fetch」是前端报错，一般是前端等后端超时：当时后端调公网 Piston 从国内访问很慢，后端一直 block，导致前端请求超时。解决方式是在 ECS 上用 Docker 自建 Piston，把 PISTON_API_URL 配成 `http://127.0.0.1:2000/api/v2`，后端只访问本机；同时把 Nginx 的 proxy_read_timeout 调大（如 60s），并写了文档记录部署和排查步骤。

**面试官**：用户随便发一段恶意代码过来执行，你怎么考虑安全？

**参考答案**：  
当前是个人/小规模使用，没有做代码审计和强隔离。Piston 本身在容器里跑，有一定隔离，但没做资源硬限（CPU/内存/时间）。生产上会加：执行超时（我们已有 piston.timeout-seconds）、限流/队列防止刷接口、可选的内存或 CPU 限制；更严格的话可以在 Piston 外再包一层沙箱或使用更隔离的执行方案，并对敏感系统调用做限制。

## 五、部署与运维

**面试官**：单 jar 是怎么打的？前端怎么一起部署？

**参考答案**：  
前端先 npm run build 打出 dist，Maven 的 resources 插件在 prepare-package 阶段把 frontend/dist 拷到 target/classes/static，再打成一个 Spring Boot fat jar，所以一个 jar 里既有后端又带前端静态资源，部署时只传这一个 jar 到服务器，用 java -jar 启动即可。

**面试官**：Docker 自建 Piston 时踩过什么坑？

**参考答案**：  
一是 Piston 官方镜像挂载的目录不能是空目录，且仓库里的 packages 是源码目录，直接挂载会报错，需要按文档处理目录结构；二是要在容器里用 ppman 安装 Java、Python 等语言，否则执行会报 runtime unknown；三是部署后要在服务器配好 PISTON_API_URL 并重启应用，否则后端仍连不到自建实例。这些都在项目文档里写了。

## 六、数据持久化与鉴权

**面试官**：管理员密码鉴权是怎么做的？为什么用 Filter？

**参考答案**：  
对写操作（POST/PUT/DELETE 等）要求请求头里带 X-Admin-Password，且与配置里的 app.admin-password 一致，否则返回 403。实现上用一个继承 OncePerRequestFilter 的 Filter，Order 设成 1 优先执行；只对 /api 开头的路径做校验，OPTIONS 预检直接放行并带上 CORS 头。用 Filter 是为了在进 Controller 之前统一鉴权，所有写接口都生效，避免在每个接口里重复判断。

**面试官**：为什么用 H2？有没有考虑过 MySQL？

**参考答案**：  
项目是个人/小规模、单机部署，H2 嵌入式、零配置、数据可持久化到文件，开发和部署都简单。若后面有多机或需要更好并发和运维工具，可以换成 MySQL，用 Spring Data JPA 的抽象，改配置和方言即可，业务代码改动不大。

## 七、难点与优化（ STAR 类问题）

**面试官**：这个项目里你遇到的最大难点是什么？怎么解决的？

**参考答案**：  
可以说「线上代码执行一直失败」：本地正常，部署后要么 Host is not specified 要么 Failed to fetch。先查日志和环境变量，确认是 Piston 地址未生效 + 公网 Piston 从国内访问超时。然后查资料决定自建 Piston，用 Docker 部署、按文档处理 packages 和语言安装，再配环境变量和 Nginx 超时，并写文档把步骤和常见错误记下来。体现定位问题、查资料、定方案、落地并沉淀文档的过程。

**面试官**：如果让你继续优化这个项目，你会做哪些？

**参考答案**：  
（按优先级）① RAG 向量持久化，用 Redis 或向量库 + 增量索引；② 代码执行加队列或限流，防止恶意刷运行占满 Piston；③ 管理员鉴权改为 JWT 或 Session，支持多用户/角色；④ 面经、题库接口加分页与缓存；⑤ 加监控与日志（如 Piston 调用失败率、接口耗时），便于线上排错。

## 九、Agent / ReAct / 技术选型与 RAG 细节

**面试官**：Agent 里 ReAct 的详细工作流程分哪几步？

**参考答案**：  
ReAct 是 Reasoning + Acting 的缩写，典型流程可以分成几步循环：  
① **Thought（思考）**：模型根据当前用户问题和已有上下文，决定下一步要做什么——是直接回答，还是需要先调用某个工具。  
② **Action（行动）**：若需要工具，模型输出要调用的工具名和参数，例如 `searchInterviews(query="字节后端", company="字节")`。  
③ **Observation（观察）**：后端执行该工具，把返回结果（如面经片段、题目链接）作为 Observation 拼回对话。  
④ **重复或结束**：模型看到 Observation 后，可以再 Thought → 再 Action（如再查一道题、再跑代码），直到模型认为信息足够，输出最终面向用户的自然语言回答。  
在我们项目里，用的是 LangChain4j 的 AiServices + Tool，底层就是这一套：用户消息进去 → 模型决定是否调 Tool → 执行 Tool 得到结果 → 结果作为下一轮输入再给模型 → 直到模型不再调工具并给出回复。没有在 Prompt 里显式写「Thought/Action/Observation」三连，但交互流程和 ReAct 一致。

**面试官**：如何防止模型陷入逻辑死循环？

**参考答案**：  
当前项目里没有显式做「防死循环」的硬限制，若要上生产，可以从几方面做：  
① **限制 Tool 调用轮数**：例如单次对话最多允许 3～5 轮 Tool 调用，超过就强制结束并返回「调用次数过多，请简化问题」之类提示。  
② **去重**：同一轮对话内，若连续两次 Action 的「工具名 + 参数」完全一致，就不再执行第二次，直接把上一次的 Observation 复用或提示「请根据已有结果回答」。  
③ **在 System Prompt 里约束**：明确写「若已通过工具得到足够信息，请直接给出最终回答，不要反复调用同一工具或重复相同操作」。  
④ **超时**：单次请求总耗时（含多轮 Tool）设上限，避免无限循环拖死服务。  
结合我们项目，算法题类请求已经短路到后端直出，不经过 Agent，本身也减少了模型乱调工具、反复查题的可能。

**面试官**：为什么用 LangChain4j，不用 Spring AI 或者 Dify/Coze？

**参考答案**：  
**选 LangChain4j 的原因**：做这个项目时，需要 RAG（向量存储、检索）+ Function Calling（Tool）一套打通，LangChain4j 的文档和示例比较全，EmbeddingStore、AiServices、Tool 注解和智谱 API 的对接都有现成用法，开发效率高；而且它是 Java 库，直接嵌在 Spring Boot 里，业务逻辑、鉴权、数据层都能自己控。  
**为什么没用 Spring AI**：当时 Spring AI 刚起步，生态和示例不如 LangChain4j 成熟，RAG 和多种模型接入的样例较少；若现在从零选型，会再对比两者在 Spring 原生集成、版本迭代上的差异。  
**为什么没用 Dify/Coze**：Dify、Coze 是低代码/无代码的 LLM 应用平台，更适合快速搭对话流、工作流，但我们要的是**代码级可控**——RAG 和面经实体、题库、代码执行接口都深度耦合在现有 Spring 应用里，需要自定义鉴权、URL 清洗、算法题短路逻辑等，用平台的话要么难以实现要么要写大量插件；所以我们选择了「Spring Boot + LangChain4j」自建，便于和现有后端统一部署、统一运维。

**面试官**：文本切分和 Embedding 模型用的是什么？为什么选这个？

**参考答案**：  
**文本切分**：我们这边没有用通用的「长文档按段切 chunk」的 DocumentSplitter。面经是结构化数据（公司、部门、岗位、内容等），每条面经对应一条可检索单元：用 `buildSearchableText` 把一条面经的字段拼成一段完整文本（如「公司: 字节\n部门: 后端\n岗位: 实习\n内容: ...」），一条面经一个 `TextSegment`，直接送去向量化。这样检索时是以「整条面经」为粒度，避免把一条面经拆成多段导致上下文割裂。若以后支持很长 PDF 文档导入，会考虑按段落或按固定 token 数切分，并加少量重叠，避免语义被截断。  
**Embedding 模型**：用的是 **AllMiniLmL6V2**，通过 LangChain4j 的 `AllMiniLmL6V2EmbeddingModel`（ONNX 本地推理）。选它的原因：① **本地部署**：不调外部 API，无配额和网络依赖，适合个人项目；② **开源、体积小**：基于 sentence-transformers 的 all-MiniLM-L6-v2，多语言效果尚可，中文面经检索够用；③ **与 LangChain4j 集成简单**：一行 Bean 即可。若数据量变大或对中文语义要求更高，可以换成智谱/OpenAI 的 Embedding API 或更大的本地模型（如 bge-large 等）。

## 十、面经 RAG 细节 / 对话记忆 / 上下文与 PE

**面试官**：面经库 RAG 过程里，chunk 过程、embedding 分别是怎么做的？

**参考答案**：  
**Chunk**：项目里没有用通用的「长文档按段切 chunk」的 DocumentSplitter。面经是结构化数据（公司、部门、岗位、内容、八股、算法题等），我按「一条面经一个可检索单元」处理：用 `buildSearchableText` 把一条面经的所有字段拼成一段文本（如「公司: 字节\n部门: 后端\n岗位: 实习\n内容: ...」），一条面经对应一个 `TextSegment`，不再做按字数/句子的切分。这样检索时是以整条面经为粒度，语义完整。  
**Embedding**：对这段文本调用 LangChain4j 的 `EmbeddingModel.embed(text).content()` 得到向量，再 `embeddingStore.add(embedding, segment)` 写入 InMemoryEmbeddingStore；检索时对 query 同样做一次 embed，用 `findRelevant(queryEmbedding, maxResults * 2, 0.5)` 做相似度检索，并可按 company/department 做元数据过滤。

**面试官**：面经 RAG 和大模型结合生成问题，和直接给大模型「面经 + 简历」输出，有做过对比吗？

**参考答案**：  
没有在项目里做严格的 A/B 对比实验。从设计上，RAG 方式的差异主要是：① **按意图检索**：用户选公司/部门或输入问题时，只检索 topK 条相关面经（相似度 + 公司/部门过滤），塞进 prompt，而不是把全库面经都塞进去；② **控制上下文长度**：检索结果会做长度截断（如单条超 800 字就截断），避免超长上下文挤占 token、影响生成质量；③ **可扩展**：面经库变大后，直接「全量面经 + 简历」会超窗口，RAG 可以固定只取最相关的几条。若要做对比，可以设计「同一公司/部门 + 同一简历」下，一组用 RAG 检索 5 条、一组用随机 5 条或全量前 N 条，比较生成问题的相关性和多样性。

**面试官**：怎么保证对话不遗忘？

**参考答案**：  
在面试模拟的会话场景（`InterviewChatService`）里，用了 **LangChain4j 的 MessageWindowChatMemory**，按 sessionId 维护一个内存里的对话窗口，例如 `withMaxMessages(20)`，即保留最近 20 轮 user/assistant 消息。每次用户发消息时，先把 UserMessage 加入 memory，再和 system prompt、本场面试问题、简历摘要、RAG 检索结果一起组成 messages 调用模型；模型返回后把 AiMessage 再加入 memory，这样下一轮就能看到历史。同时会把每轮 user/assistant 持久化到数据库（`InterviewChatMessage`），便于历史会话回放；但当前轮对话的「记忆」依赖的是内存里的 MessageWindowChatMemory，没有从 DB 按 session 恢复历史消息到 memory，所以会话结束后再开新会话就是新的 memory。若要「长期不遗忘」，可以在恢复会话时从 DB 拉取该 session 的历史消息灌回 memory，或只保留最近 N 条以控制长度。

**面试官**：上下文过长怎么办？有约束吗？

**参考答案**：  
有做长度约束，但没有按 token 精确卡上限。具体：① **RAG 检索结果**：每条面经文本超过一定字数就截断，例如深挖问题生成用 800 字、会话里的 RAG 参考用 500 字、复盘用 600 字，避免单条过长；② **简历**：在拼进 system 时，简历超过 2000 字会截断为 2000 字 + "..."，存库时也有 5000 字的截断；③ **对话轮数**：MessageWindowChatMemory 只保留最近 20 条消息，超过的会被挤掉。没有在 prompt 里显式写「总 token 不超过 X」，后续如果要严格控 token，可以按模型上下文窗口算一下，对 system + 历史消息 + RAG 做整体截断或摘要。

**面试官**：PE（Prompt Engineering）过程里，做了哪些 prompt 约束？

**参考答案**：  
主要在做格式和规则约束，避免模型乱输出。例如：① **深挖问题生成**（InterviewCoach）：要求「直接以 1. 2. 3. 开头、禁止开场白和总结、每题 50 字以内、顺序为实习→项目→八股→大模型」；② **复盘**（ReplayCoach）：要求「不用 *、**、###，用数字序号+小标题+冒号分段，列表用 (1)(2)(3) 或一二三」；③ **带 Tool 的助手**（AssistantWithTools）：在 SystemMessage 里明确「工具返回的 URL 必须原样输出、禁止包 HTML、禁止改 leetcode.com 路径、必须输出链接」等，减少链接被改成 HTML 或错误路径；④ **答疑角色**：要求「答案 300 字以内、直接给要点、不要冗长开场」。没有用 few-shot 或复杂的 chain-of-thought 提示，主要是「角色 + 格式 + 禁止项」这类约束。

**面试官**：为什么用 LangChain4j？Tool 怎么调用的？有几个 Tool？

**参考答案**：  
**为什么用 LangChain4j**：当时需要 RAG（向量存储、检索）+ Function Calling（Tool）一套打通，LangChain4j 的文档和示例全，EmbeddingStore、AiServices、智谱对接都方便，且是 Java 库，能嵌在 Spring Boot 里，业务和鉴权都能自己控。  
**Tool 怎么调用**：在 `InterviewAssistantTools` 里用 `@Tool("描述")` 注解在方法上，描述里写清楚「何时调用、参数含义」；在 `InterviewAgentWithToolsService` 里用 `AiServices.builder(AssistantWithTools.class).chatLanguageModel(chatModel).tools(tools).build()` 把 ChatModel 和 tools 绑在一起，用户发消息后调用 `getAssistant().chat(message)`，LangChain4j 内部会根据模型返回的 tool_calls 解析出工具名和参数，反射调用对应方法，再把结果作为 observation 塞回对话，由模型生成最终回复。  
**有几个 Tool**：一共 5 个——① searchInterviews：按公司/部门/关键词检索面经；② listAlgorithmQuestions：列题库或随机一道题；③ findAlgorithmQuestionByTitle：按题名查题（含联网搜原题）；④ getAlgorithmQuestionById：按 ID 取题；⑤ runCode：执行代码并返回 stdout/错误。
